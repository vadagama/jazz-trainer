# 02 — Audio / MIDI / Metronome engine

## Цель документа

Выбрать и обосновать подход к воспроизведению звука и таймингу метронома, а также
зафиксировать архитектуру transport/scheduler, которая позже масштабируется до
полноценного accompaniment engine (бас, гармония, мелодия, барабаны, MIDI-экспорт).

---

## 1. Проблема точного тайминга в браузере

`setInterval`/`setTimeout` **непригодны** для музыкального тайминга: их колбэки
вызываются main-потоком JS, который подвержен джиттеру (GC, рендер, layout), а гранулярность
таймеров — десятки миллисекунд. На слух дрожание в 10–30 мс уже разрушает грув.

Правильный подход — **look-ahead scheduling** (паттерн Криса Уилсона, «A Tale of Two
Clocks»):

- Звуковые события планируются заранее на точной шкале `AudioContext.currentTime`
  (аппаратные часы аудиокарты, sample-accurate).
- Низкочастотный таймер (`setInterval` ~25 мс) лишь периодически «заглядывает вперёд» на
  окно ~100 мс и планирует все события, попадающие в это окно, через
  `oscillator.start(time)` / `source.start(time)`.
- Даже если JS-таймер дёрнулся, события уже запланированы в аудиодвижке и прозвучат точно.

Именно этот паттерн должен лежать в основе метронома.

## 2. Сравнение вариантов

| Критерий | Raw Web Audio API | **Tone.js** | Web MIDI API |
|---|---|---|---|
| Точность тайминга | ★★★ (сами пишем look-ahead) | ★★★ (look-ahead встроен в Transport) | n/a (только выход на внешний синт) |
| Объём своего кода | высокий (свой clock, scheduler) | низкий | средний |
| Музыкальная модель времени | нет (секунды) | есть (`"4n"`, `"1m"`, bars:beats:sixteenths) | нет |
| Транспорт (play/pause/stop/loop) | пишем сами | встроен (`Tone.Transport`) | пишем сами |
| Встроенные синты/сэмплеры | нет | есть (`Synth`, `Sampler`, `MembraneSynth`…) | нет (звук во внешнем устройстве) |
| Путь к accompaniment | всё руками | готовые инструменты + scheduler | нужен внешний звук/SF |
| Зависимости | 0 | 1 (Tone.js, ~автономна) | 0 (браузерный API) |
| Зрелость/поддержка | браузерный стандарт | широко используется, активна | поддержка зависит от браузера |

### Почему не raw Web Audio
Дало бы максимум контроля и ноль зависимостей, но потребовало бы вручную написать
clock, look-ahead scheduler, музыкальную модель времени и транспорт — то есть
переизобрести существенную часть Tone.js. Для MVP это лишний риск и объём.

### Почему не Web MIDI (как основной движок)
Web MIDI **не производит звук** — он отправляет MIDI-сообщения на внешний синтезатор/DAW.
Поддержка в браузерах неравномерна, и требовать у пользователя внешний инструмент для
метронома неприемлемо. Web MIDI остаётся как **будущий опциональный выход** (см. §6).

### Почему Tone.js (выбор)
- `Tone.Transport` уже реализует look-ahead scheduling поверх `AudioContext` — точный
  тайминг «из коробки».
- Музыкальная модель времени (`"4n"`, bars:beats:sixteenths, BPM, тактовый размер) точно
  соответствует нашей доменной модели `bar/beat/subdivision/tick`.
- Встроенные `Synth`/`Sampler`/`MembraneSynth` дают звук метронома сразу и инструменты
  для будущего accompaniment без новой инфраструктуры.
- `Transport.scheduleRepeat` / `schedule` — готовый event scheduler, на который встанут
  все будущие партии.
- Изолируется за нашим интерфейсом, поэтому при необходимости заменяема.

**Решение: Tone.js поверх Web Audio API для MVP, с заложенным путём к Web MIDI и
MIDI-экспорту.**

## 3. Музыкальная модель времени

Доменная модель в `packages/music-core` (фреймворк-независимая):

```ts
// размер такта
interface TimeSignature { beatsPerBar: number; beatUnit: 4 | 8; } // 4/4, 6/8, ...

// позиция в музыкальном времени
interface MusicalPosition { bar: number; beat: number; tick: number; } // tick: 0..PPQ-1

const PPQ = 480; // pulses per quarter note — внутреннее разрешение
```

Конвертации (чистые функции, 100% тестируемы):
- `positionToTicks(pos, sig) -> number`
- `ticksToPosition(ticks, sig) -> MusicalPosition`
- `ticksToSeconds(ticks, bpm) -> number` и обратно
- учёт `beatUnit` (в 6/8 доля = восьмая, группировка 3+3)

Эта модель — единый язык времени для метронома и будущих партий. Tone.js используется
как **исполнитель** этой модели, а не как её источник правды.

## 4. Архитектура transport + scheduler + инструменты

```
            ┌─────────────────────────────────────────────┐
            │  TransportEngine (обёртка над Tone.Transport) │
            │  bpm · timeSignature · position · status      │
            │  play() pause() stop() seekToBar(n)           │
            │  onTick(cb)  // bar/beat события для UI        │
            └───────────────┬───────────────────────────────┘
                            │ look-ahead scheduling
            ┌───────────────┴───────────────┐
            ▼                                ▼
   ┌──────────────────┐            ┌──────────────────────┐
   │ MetronomeInstrument│          │ (future) BassInstrument │
   │ schedule(window)   │          │ CompInstrument          │
   │ клики по активным  │          │ DrumInstrument          │
   │ долям, strong/weak │          │ MelodyInstrument        │
   └──────────────────┘            └──────────────────────┘
            общий интерфейс Instrument { prepare(ctx); schedule(window, transport); dispose(); }
```

- **TransportEngine** — единственный владелец музыкального времени. Инкапсулирует
  `Tone.Transport`, публикует позицию (bar/beat) наружу через колбэк/эмиттер.
- **Instrument** — интерфейс: получает окно времени и транспорт, планирует свои события.
  Метроном — первая реализация.
- **PlaybackStateMachine** — чистый редьюсер (без Tone): статус (idle/playing/paused),
  currentBar, currentBeat, selectedBar; реагирует на команды и на tick-события транспорта.
  Тестируется без звука.

### Метроном: требования и реализация
- BPM (глобальная настройка пользователя).
- Размеры: 4/4, 3/4, 2/4, 5/4, 6/8 + расширяемая модель.
- play / pause / stop (stop → возврат в начало) / next bar / prev bar / select bar.
- Клик только на выбранных долях (маска активных долей).
- Разный звук клика для сильной и слабой доли (strong/weak).
- При Play — старт с выбранного такта или с начала; при Pause — позиция сохраняется;
  при Stop — возврат в начало формы.

Звук клика — короткие синтезированные импульсы (`Tone.MembraneSynth`/короткий
`Synth`/шумовой клик); сильная доля выше/громче. Сэмплы можно подключить позже —
интерфейс инструмента не меняется.

### Разделение глобального и пер-сетка
- **Глобально (на пользователя)**: BPM, тип/звук клика, громкость, дефолтные настройки.
- **Пер-сетка**: тактовый размер (`TimeSignature`) — часть данных гармонической сетки.
- TransportEngine при загрузке сетки берёт размер из сетки, остальное — из UserSettings.

## 5. Граница «чистое ядро / звуковой адаптер»

```
music-core/
  time/            ← чистое: модель времени, конвертации        (unit-тест без Tone)
  playback/        ← чистое: PlaybackStateMachine               (unit-тест без Tone)
  audio/
    TransportEngine.ts   ← адаптер над Tone.Transport
    MetronomeInstrument.ts
    instrument.ts        ← интерфейс Instrument
```

В тестах ядра Tone.js мокается; проверяются конвертации времени и переходы state
machine. Звуковой адаптер тестируется отдельно (smoke) и проверяется вручную/в e2e.

## 6. Путь к будущему (заложено, но не реализуется в MVP)

- **MIDI event model**: внутренний нейтральный формат
  `{ tick, durationTicks, pitch, velocity, channel }`. Метроном и будущие генераторы
  пишут в этот формат; рендер в звук — отдельный шаг.
- **Accompaniment-генераторы** (walking bass, comping, arpeggios, drums, guide tones)
  читают **chord timeline** сетки и музыкальную модель времени, выдают MIDI-события,
  которые исполняет соответствующий Instrument.
- **Web MIDI выход**: опциональный backend, отправляющий те же MIDI-события на внешнее
  устройство вместо/вместе с Tone-синтом.
- **MIDI-экспорт (.mid)**: сериализация MIDI event model в стандартный файл (вне MVP).

Эти возможности не входят в MVP, но ни одна из них не потребует переписывать transport,
time-model или UI — только добавить новые реализации `Instrument` и генераторы.

---

## 7. Зависимости и решения

- Зависимость: `tone` (в `apps/web`; ядро транспорта — в `music-core`, Tone подключается
  на стороне web-сборки/адаптера, чтобы `music-core` оставался тестируемым в Node).
- `AudioContext` запускается только после пользовательского жеста (политика автоплея
  браузеров) — старт транспорта инициируется кликом Play.
- Тайминг-критичный код никогда не зависит от React-рендера: UI читает позицию из стора,
  обновляемого tick-событиями, но звук планируется в аудиодвижке заранее.
