# 08 — Architectural Decision Records (ADR)

Зафиксированные решения и допущения. Формат: контекст → решение → следствия.

---

## ADR-001 — Monorepo на npm workspaces
**Контекст.** Нужно изолировать музыкальную логику и общие типы от web/api, но без
оверинжиниринга для MVP.
**Решение.** Monorepo (`apps/web`, `apps/api`, `packages/music-core`, `packages/shared`)
на **npm workspaces** без Turborepo/Nx.
**Следствия.** Чистые границы и переиспользование кода; простые корневые скрипты; при
росте можно добавить Turborepo без смены структуры.

## ADR-002 — Audio: Tone.js поверх Web Audio API
**Контекст.** Метроном требует точного тайминга (не setInterval); архитектура — под
будущий accompaniment engine.
**Решение.** **Tone.js** (look-ahead scheduler, музыкальная модель времени, встроенные
синты) как исполнитель доменной time-model. Raw Web Audio и Web MIDI отклонены как
основной движок (см. [02-audio-engine.md §2](02-audio-engine.md)).
**Следствия.** Быстрый точный метроном; готовый transport/scheduler; Web MIDI и
MIDI-экспорт остаются опциональным будущим выходом без переписывания ядра.

## ADR-003 — Auth: Google OAuth + dev-login fallback (опциональная, не обязательная)
**Контекст.** Google OAuth требует реальных credentials; проект должен запускаться и
тестироваться без них. Приложение **public-first** (ADR-010): auth не обязательна для входа.
**Решение.** Полный Google OAuth + **dev-login** (`AUTH_DEV_MODE=true`, выключен в проде).
Auth — аддитивная: `GET /auth/me` возвращает `user|null`, анонимный доступ разрешён.
**Следствия.** e2e и локальный запуск без Google Cloud; изоляция по userId одинакова в
обоих режимах; в проде dev-login отключён; гость пользуется публичной частью без входа.

## ADR-004 — Backend: Fastify + Drizzle + better-sqlite3
**Контекст.** Лёгкий типобезопасный REST на SQLite с миграциями.
**Решение.** **Fastify** (HTTP), **Drizzle ORM + better-sqlite3** (БД), **drizzle-kit**
(миграции).
**Следствия.** Первоклассный TS, лёгкие зависимости, простые миграции; SQLite-файл
конфигурируется через env.

## ADR-005 — Хранение сетки как JSON внутри HarmonyGrid
**Контекст.** Такты/аккорды — вложенная структура; для MVP важна простота.
**Решение.** `content` — JSON-поле, валидируемое `GridContentSchema` (Zod) из `shared`;
размер сетки дублируется колонкой `time_signature`.
**Следствия.** Атомарные save/load, прямой round-trip с DSL, лёгкая эволюция через
`version`; минус — нет SQL по отдельным аккордам (не нужно в MVP; путь к нормализации без
смены публичного API — [03-data-model.md §6](03-data-model.md)).

## ADR-006 — Единый источник контрактов в `packages/shared`
**Контекст.** Рассинхронизация типов между фронтом и бэком — частый источник багов.
**Решение.** Все DTO и Zod-схемы — в `@jazz/shared`, импортируются и api, и web.
**Следствия.** Одна точка правды для валидации и типов.

## ADR-007 — React = view + actions; музыкальная логика только в `music-core`
**Контекст.** Требование ТЗ: не размазывать музыкальную логику по компонентам.
**Решение.** Вся музыкальная логика — в `music-core`; React читает состояние из
сторов/квери и шлёт действия; тайминг-критичный код не на пути рендера.
**Следствия.** Тестируемость ядра без DOM; переносимость логики на бэк; чистый путь к
accompaniment.

## ADR-008 — Поэтапная разработка: фича = ветка → MR, тесты на каждом шаге
**Контекст.** Требование поэтапности с прогоном тестов, чтобы ничего не разваливалось.
**Решение.** Каждая фича — `feature/<name>` → PR в `main`; на каждой фиче полный прогон
тестов; обновление документации. «MR» = PR (gh CLI), т.к. remote уже настроен.
**Следствия.** Прослеживаемость, регрессионная защита, актуальная документация.

## ADR-009 — Сессии через подписанные cookie (server sessions)
**Контекст.** Нужна простая безопасная auth с изоляцией данных.
**Решение.** Серверные сессии (таблица `sessions`) + HttpOnly cookie `sid`; в dev —
same-site через Vite proxy.
**Следствия.** Нет хранения токенов на клиенте; logout инвалидирует сессию; готово к
проду (Secure cookie).

## ADR-010 — Public-first доступ; auth добавляет персональные возможности
**Контекст.** Требование: вход в приложение не покрыт аутентификацией по умолчанию. По
руту `/` — публичный каталог гармонических сеток (список + плеер с навигацией). Публичная
часть позволяет искать сетки и менять настройки метронома/размер/тональность. Персональное
(свой каталог, сохранение настроек, профиль, лайки, редактирование/создание, импорт DSL)
— только после входа.
**Решение.**
- API делится на **публичные** (без auth: публичный каталог `/grids/public[/:id]`,
  `/patterns`, `/generate`, `/auth/me`) и **приватные** (auth: `/grids/mine`, CRUD своих,
  copy, import, `/settings`, лайки). `optional-auth` на всех, `require-auth` на приватных.
- `harmony_grids.visibility` (`public|private`); публичный каталог в MVP — сетки **system
  user** (seed). Пользователи **копируют** публичные себе (`source_grid_id`), но **не
  публикуют** свои (задел на будущее).
- **Анонимные настройки** метронома — в **localStorage** (`useLocalSettingsStore`);
  авторизованные — на сервере (`user_settings`). `useEffectiveSettings` выбирает источник;
  при входе предлагается перенести локальные в профиль.
- Смена анонимом размера/тональности/BPM публичной сетки — **локальный override
  плейбека**, не сохраняется (публичные данные read-only).
- **Лайки** (`grid_likes`) — только авторизованный, только по публичной сетке.
**Следствия.** Низкий порог входа; персонализация — поверх. Чуть сложнее
permission-модель (public vs private), покрыта явными тестами. Публикация
пользовательских сеток и переход настроек на серверные при входе — реализуемы без смены
контракта.

## Открытые допущения (фиксируются здесь, уточняются при необходимости)
- DnD в сетке (dnd-kit) в MVP — минимально (перестановка), полная функциональность позже.
- Звук клика — синтезированный (Tone), качественные сэмплы — не в MVP.
- `random diatonic` генератор принимает seed для детерминированных тестов.
- Светлая тема — опционально после MVP (dark-first).
