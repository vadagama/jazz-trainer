# 01 — Архитектура системы

## Цель документа

Зафиксировать общую архитектуру Jazz Trainer: слои, потоки данных, границы
ответственности и принципы, которые гарантируют, что музыкальная логика не размажется
по React-компонентам, а приложение будет готово к будущему accompaniment engine.

---

## 1. Высокоуровневый обзор

Jazz Trainer — браузерный тренажёр джазовой гармонии. Приложение **публичное по
умолчанию**: без входа доступны публичный каталог сеток, плеер с метрономом и генерация.
Аутентификация **добавляет** персональные возможности (свой каталог, сохранение настроек,
профиль, лайки, редактирование своих сеток и импорт DSL).

Пользователь тренируется под точный метроном, импортирует/экспортирует гармонию через
DSL и генерирует прогрессии по паттернам.

### Модель доступа (public-first)
- **Аноним**: чтение публичного каталога (поиск/плеер), генерация, локальная смена
  BPM/клика/размера/тональности (localStorage + ephemeral override) — без записи в БД.
- **Авторизованный** (доп. поверх): `GET /grids/mine`, CRUD своих сеток, копирование
  публичной «себе», сохранение настроек на сервере, лайки, профиль.
- Детали — [03-data-model.md](03-data-model.md) §1/§4, [04-api.md](04-api.md) §1.1/§6,
  [05-frontend.md](05-frontend.md) §2/§4a.

Система состоит из четырёх слоёв (monorepo, npm workspaces):

```
┌──────────────────────────────────────────────────────────────┐
│  apps/web  (React + Vite)                                      │
│  ── view-слой: рендерит состояние, шлёт действия ──            │
│   pages · components · stores(Zustand) · queries(TanStack)     │
└───────────────┬───────────────────────────────┬───────────────┘
                │ REST (cookie-сессия)           │ импорт пакета
                ▼                                 ▼
┌──────────────────────────────┐   ┌────────────────────────────┐
│  apps/api  (Fastify)         │   │  packages/music-core        │
│  routes · services · auth    │   │  DSL · chord-model ·        │
│  Drizzle ORM · SQLite        │   │  transport · generator ·    │
│                              │   │  playback-state-machine     │
└───────────────┬──────────────┘   └──────────────┬─────────────┘
                │ импорт пакета                    │ импорт пакета
                ▼                                  ▼
        ┌──────────────────────────────────────────────┐
        │  packages/shared                              │
        │  типы DTO · Zod-схемы · доменные enum'ы        │
        └──────────────────────────────────────────────┘
```

Ключевой принцип: **зависимости направлены внутрь**. `web` и `api` зависят от
`music-core` и `shared`; `music-core` зависит только от `shared`; `shared` ни от чего.
Музыкальная логика не знает ни про React, ни про Fastify, ни про БД.

## 2. Назначение слоёв

### 2.1 `packages/shared`
Чистые типы и контракты, общие для фронта и бэка.
- TypeScript-типы DTO (`HarmonyGridDTO`, `UserSettingsDTO`, `UserDTO`, …).
- **Zod-схемы** как единственный источник правды для валидации (в т.ч. JSON-контент
  сетки `GridContentSchema`). Бэк валидирует вход, фронт — формы (RHF + Zod).
- Доменные enum'ы и константы (поддерживаемые размеры, типы клика, тональности).
- Никакой логики, никаких зависимостей от рантайма (без Fastify/React/Tone).

### 2.2 `packages/music-core`
Вся музыкальная логика — **фреймворк-независимая**, тестируется в изоляции.
- **DSL parser / serializer** — текст ↔ структура гармонии.
- **Chord model** — разбор символа аккорда (`Dm7b5`) в структуру (root, quality, extensions).
- **Harmony generator** — data-driven генерация прогрессий по паттернам.
- **Time model** — `bar / beat / subdivision / tick`, конвертации, размеры.
- **Transport engine** — обёртка над Tone.js Transport (play/pause/stop, look-ahead scheduling).
- **Metronome scheduler** — планирование кликов по активным долям.
- **Playback state machine** — чистый редьюсер состояния воспроизведения
  (current bar/beat, selected bar, transport status). Тестируется без звука.

`music-core` делится на «чистое ядро» (без Tone.js: model, parser, generator, state
machine — 100% unit-тестируемо) и «звуковой адаптер» (Tone.js: transport, metronome —
изолирован за интерфейсом, мокается в тестах). См. [02-audio-engine.md](02-audio-engine.md).

### 2.3 `apps/api`
Лёгкий REST-бэкенд на Fastify.
- **routes** — HTTP-эндпоинты, валидация вход/выход через Zod-схемы из `shared`.
- **services** — бизнес-логика (CRUD сеток, копирование, генерация через `music-core`).
- **auth** — Google OAuth + dev-login fallback, cookie-сессии.
- **db** — Drizzle ORM + better-sqlite3, миграции drizzle-kit.
- **изоляция по userId** — сквозной guard: каждый запрос к данным фильтруется по
  владельцу; чужие сетки/настройки недоступны. См. [04-api.md](04-api.md).

### 2.4 `apps/web`
React-фронтенд — **view + actions**, без музыкальной логики.
- **pages** — Login, Dashboard, Editor/Trainer, Settings.
- **components** — UI (shadcn/ui + Tailwind), кастомный CSS Grid компонент сетки.
- **stores (Zustand)** — клиентское состояние: transport/playback UI-проекция, editor.
- **queries (TanStack Query)** — серверное состояние (сетки, настройки): кэш, мутации.
- **engine-bridge** — тонкий слой, связывающий React-сторы с `music-core` transport
  (подписка на события «текущий такт/доля», отправка команд play/pause/...).

## 3. Потоки данных

### 3.1 Загрузка и редактирование сетки
```
Публичное чтение (без auth):
web ──GET /api/grids/public[/:id]──▶ api/service ──Drizzle(WHERE visibility='public')──▶ SQLite

Своя сетка (auth):
web ──GET /api/grids/mine|/:id──▶ api/service ──Drizzle(WHERE user_id=current)──▶ SQLite
       ▲                                                    │
       └──────────── HarmonyGridDTO (JSON content) ─────────┘
web парсит/сериализует content через music-core DSL при импорте/экспорте.
Редактирование своей сетки меняет editor-store; сохранение — PATCH /api/grids/:id.
Публичная сетка read-only: «изменения» — локальный playbackOverride или copy «себе».
```

### 3.2 Воспроизведение (метроном по сетке)
```
UI команда (Play) ──▶ engine-bridge ──▶ music-core Transport (Tone.js)
                                              │ look-ahead scheduling
                                              ▼
                            metronome scheduler планирует клики
                                              │ tick-события (bar/beat)
                                              ▼
   playback-state-machine ──▶ Zustand store ──▶ React подсвечивает текущий такт/долю
```
Звук планируется в `music-core`/Tone.js, React лишь отражает события — UI-рендер
никогда не находится на «горячем пути» тайминга.

### 3.3 Генерация гармонии
```
web (выбор паттерна/тональности/длины) ──POST /api/generate──▶ api
api ──▶ music-core generator ──▶ структура гармонии ──▶ DTO ──▶ web
web показывает результат в редакторе; пользователь правит и сохраняет.
```
Генератор живёт в `music-core` и вызывается с бэка (можно и с фронта — один и тот же
пакет), что исключает дублирование логики.

## 4. Принцип «React = view + actions»

- Компоненты **не** парсят аккорды, **не** считают тайминг, **не** содержат музыкальных
  правил. Они читают состояние из сторов/квери и вызывают действия.
- Любая музыкальная операция = вызов функции из `music-core`.
- Состояние воспроизведения — это проекция `playback-state-machine` в Zustand-стор;
  компоненты подписаны на стор и перерисовываются по изменениям.
- Это даёт: тестируемость ядра без DOM, переносимость логики на бэк, и чистый путь к
  accompaniment engine (новые «инструменты» добавляются в `music-core`, а не в UI).

## 5. Фундамент под будущий accompaniment engine

Метроном в MVP — это **первый инструмент** поверх общего transport. Архитектура заранее
закладывает:

- **Transport engine** — единый источник музыкального времени (BPM, позиция, размер),
  на который подписываются все «инструменты».
- **Event scheduler** — look-ahead планирование событий на временной шкале (сейчас —
  клики метронома; позже — ноты баса/гармонии/барабанов).
- **Musical time model** — `bar / beat / subdivision / tick` уже сейчас, чтобы будущие
  партии адресовались в музыкальном времени, а не в секундах.
- **Chord timeline** — гармоническая сетка уже представлена как последовательность
  аккордов во времени; accompaniment-генераторы будут читать её как вход.
- **MIDI event model** — внутренний формат событий (note on/off, time, velocity, channel),
  нейтральный к источнику звука. MVP проигрывает через встроенные звуки (Tone.js synth),
  но модель готова к Web MIDI и экспорту .mid.
- **Инструмент как интерфейс** — `Instrument { schedule(events, transport) }`. Метроном
  реализует этот интерфейс; бас/гармония/барабаны добавятся как новые реализации без
  изменения transport и UI.

Подробности — в [02-audio-engine.md](02-audio-engine.md).

## 6. Границы и правила, которые нельзя нарушать

1. `shared` не импортирует ничего из рантайм-фреймворков.
2. `music-core` не импортирует React, Fastify, Drizzle.
3. UI-компоненты не содержат музыкальной логики — только вызовы `music-core` и работу со
   стором/квери.
4. Валидация — через единые Zod-схемы из `shared` (один источник правды для фронта и бэка).
5. Доступ к данным всегда проходит через guard изоляции по userId на бэке.
6. Звуковой адаптер (Tone.js) изолирован за интерфейсом и мокается в тестах ядра.

## 7. Тестовая стратегия по слоям

| Слой | Что тестируем | Инструмент |
|---|---|---|
| `shared` | Zod-схемы (валид/невалид) | Vitest |
| `music-core` | parser, serializer, round-trip, generator, time-model, state machine | Vitest (Tone замокан) |
| `api` | эндпоинты, permission-изоляция | Vitest + Supertest |
| `web` | ключевые компоненты/сторы | Vitest + Testing Library |
| e2e | сценарий «сетка → DSL → метроном → переключение тактов» | Playwright |

Подробная разбивка — в [07-features.md](07-features.md).
