# Solution

## Mode

`architecture`

## Context

Барабанная «клетка» (`DrumCell`) сейчас описана как **пул + правила расписания**:
`grooveMoleculePool` играет каждый такт слоями, `variation*`/`fill*` вставляются по
модульным правилам. Это компактно, но не позволяет явно расставить молекулы по тактам.
Пользователь хочет, чтобы клетка (по аналогии с молекулой — сеткой долей) стала
**сеткой тактов**, где молекулы размещаются как клипы со спанами: от начала до конца,
в середине, брейки, со середины до конца и т.д. Плюс общий уровень velocity клетки.
Цель итерации — спроектировать модель данных, детерминированную сборку в движке,
UX редактора и путь миграции, не сломав идемпотентность `assembleBar`.

## Input Summary

- Промпт: клетка = таймлайн тактов с размещением молекул (спаны start/length), общий
  velocity клетки, плюс другие улучшения.
- Контекст архитектуры (иерархия atoms→molecules→cells→organisms, текущая pool-модель,
  требование идемпотентности `assembleBar`, известные ограничения: `bars:2` не поддержан,
  `settingsComplexity`=2 захардкожен, full-form организмы `weight:0` не играют) — сохранён
  в `input/prompt.md`.

## Proposed Solution

Ввести **аранжировочную (таймлайн) модель** клетки: новое поле `arrangement: DrumClip[]`,
где клип = `{ moleculeId, startBar, lengthBars, mode, velocityMul, probability, lane }`.
Движок для каждого такта детерминированно выбирает клипы, активные на `barInCell`
(`startBar ≤ barInCell < startBar+lengthBars`), тайлит молекулу (1 или 2 такта) по спану и
эмитит атомы, масштабируя velocity на `cell.velocity × clip.velocityMul × dynMul`.
Идемпотентность сохраняется: результат зависит только от `(cell, barInCell, barInOrganism,
seed)`; вероятностные клипы сидируются по абсолютному такту. Миграция — **аддитивная и
двухрежимная**: движок использует `arrangement`, если он задан, иначе старый pool-путь;
конвертер превращает 20 текущих клеток в аранжировки (groove-клипы на всю длину, fill/
variation-клипы в позициях расписания). Редактор — таймлайн-грид «такты × лейны», клик по
клипу открывает таблицу молекулы (существующий `MoleculeTable`). Заодно чинится `bars:2`
(естественно ложится на тайлинг) и появляется явное `velocity` клетки.

## Key Decisions

- **`arrangement: DrumClip[]` как новый источник правды** — явная расстановка выразительнее
  и предсказуемее, чем модульные правила; напрямую отражает «в этом такте играет вот это».
- **Двухрежимный движок (arrangement ∥ legacy pool)** — снижает риск: старый путь и тесты
  живут, пока все клетки не мигрированы; переключение по наличию `arrangement`.
- **Клип несёт `mode: 'layer' | 'replace'`** — `replace` реализует брейки/сбивки, которые
  заглушают грув на своём диапазоне; `layer` — накладывается (как сейчас fill).
- **`cell.velocity` (0..1) — музыкальный мастер-уровень**, отдельно от `volumeMul`
  (микшерный). Атом-velocity × cell.velocity × clip.velocityMul × dynMul.
- **Тайлинг молекулы по спану чинит `bars:2`** — `molBar = localBar % molecule.bars`,
  сдвиг `atTick` в текущий такт; 2-тактовые молекулы наконец играбельны.
- **Вероятность клипа сидируется по `barInOrganism`**, а не по «настоящему» random —
  иначе ломается идемпотентность (планировщик пересобирает такт несколько раз).
- **Конвертер legacy→arrangement** — миграция без потерь и A/B-сверка звука.

## Details

### Components and Responsibilities

- **`drumPatternTypes.ts`** — новые типы `DrumClip`, `ClipMode`, расширение `DrumCell`
  (`arrangement?`, `velocity`), опциональный `dynamics.envelope`. Legacy-поля становятся
  optional (переходный период).
- **`DrumPatternEngine.assembleBar`** — развилка: `arrangement` → таймлайн-сборка (активные
  клипы по `barInCell`, тайлинг, replace-подавление лейнов), иначе legacy pool-путь.
  Остаётся чистой/идемпотентной.
- **`drumCells.ts`** — определения клеток переезжают на `arrangement`; добавляется
  `convertLegacyCell(cell): DrumCell` для одноразовой миграции и рантайм-фоллбэка.
- **`admin-drum-constructor` (CellEditor)** — заменяется на таймлайн-редактор (грид тактов ×
  лейны, клипы-спаны, инспектор с cell velocity/dynamics). Клик по клипу → `MoleculeTable`.
- **`useDrumPreview`** (хост) — без изменений контракта; играет собранные `DrumHit[]`.
  Возможна подсветка playhead по тактам (доп. улучшение).

### Integrations

- **Движок → планировщик** (`drumInstrument.schedule`): без изменений сигнатуры — `assembleBar`
  по-прежнему возвращает `DrumHit[]` по такту. Инициатор — планировщик (sync, pull).
- **Плагин Конструктор → хост-preview**: через `usePluginDrumPreview` (инъекция из
  `PluginProvider`), без изменений. Плагин собирает `DrumHit[]` через `DrumPatternEngine` с
  `moleculeOverrides`.
- **Организмы → клетки**: организм по-прежнему выбирает клетку на секцию; аранжировка живёт
  внутри клетки. Открытый вопрос — не поглощает ли таймлайн уровень организма (см. ниже).

### Data Ownership

- Каноничные молекулы/клетки — `music-core` (const-реестры, **не мутируются**). Конструктор
  работает на локальных копиях; правки атомов передаются в движок как `moleculeOverrides`.
- `arrangement` — часть определения клетки в `music-core`. Пользовательские правки в
  Конструкторе — in-memory (preview-only, как сейчас); персистентности пока нет.

### Runtime Behavior

Таймлайн-сборка одного такта (детерминированно):

```
dynMul = dynamics(barInCell, length)
suppressedLanes = ∅
for i, clip in cell.arrangement:              // стабильный порядок = порядок массива
  if !(startBar ≤ barInCell < startBar+lengthBars): continue
  if clip.probability < 1:
     r = seeded(seed, barInOrganism, i)()      // idempotent
     if r > clip.probability: continue
  localBar = barInCell - clip.startBar
  molBar   = localBar % molecule.bars          // тайлинг 1/2-тактовых
  emit molecule atoms где floor(atTick / barTicks) == molBar,
       atTick -= molBar*barTicks,              // сдвиг в текущий такт (фикс bars:2)
       vel = atom.vel * cell.velocity * clip.velocityMul * dynMul
  if clip.mode == 'replace': suppressedLanes += clip.lane
drop hits чьих lane ∈ suppressedLanes (кроме самих replace-клипов)
apply swing → resolve articulations → drop vel≤0
```

Идемпотентность: выход зависит только от `(cell, barInCell, barInOrganism, seed)`.
Порядок клипов в массиве — единственный источник приоритета (без скрытого random).

### Deployment and Operations

- Чистые изменения в `music-core` + `admin-drum-constructor`; сборки/инфры не затрагивают.
- Прогон `npm run typecheck && lint && test`; расширить набор тестов `assembleBar`
  (тайлинг, replace-подавление, вероятность-идемпотентность, границы спана).
- Фичефлаг не обязателен (двухрежимность по данным), но можно завести флаг «timeline cells»
  на время миграции контента.

### Security and Compliance

- Не применимо (клиентская музыкальная логика, без PII/сети). RBAC уже гейтит раздел
  (`content:write`).

### Quality Attributes

- **Детерминизм/идемпотентность** — обязательное свойство, сохраняется (см. Runtime).
- **Производительность** — сборка O(clips) на такт вместо O(pool); порядок тот же, клипов
  единицы-десятки. Без регрессий.
- **Обратная совместимость** — legacy-путь сохранён до полной миграции 20 клеток.
- **Расширяемость** — клипы легко несут новые поля (probability, envelope) без слома API.
- **Тестируемость** — явная расстановка проще тестируется, чем модульные правила.

### Options and Trade-offs

- **A. Полная замена pool → timeline (сразу).** Плюс: одна модель, чистый код. Минус:
  большой единовременный рефактор движка + редактора + миграция 20 клеток; риск регресса.
- **B. Аддитивно, двухрежимно (РЕКОМЕНДАЦИЯ).** `arrangement?` + фоллбэк на pool + конвертер;
  мигрируем контент постепенно, потом депрекейтим legacy. Плюс: низкий риск, A/B-сверка.
  Минус: временно две ветки сборки в движке.
- **C. Гибрид: pool как «бэкбон грува» + тонкий overlay размещения только для fill/break/
  секций.** Плюс: минимальный объём. Минус: не даёт полной выразительности («со середины до
  конца» для любой молекулы), остаётся два ментальных механизма.

Выбор: **B** — двигаемся к таймлайну как к единой модели, но безопасно.

### Additional Improvements (сверх запроса)

1. **Фикс `bars:2`** — тайлинг делает 2-тактовые молекулы играбельными (сейчас латентный баг).
2. **`mode: replace` для брейков** — брейк заглушает грув на своём такте/лейне.
3. **`probability` на клипе** — «иногда» играемые сбивки/госты, детерминированно по такту.
4. **Типизированные лейны** (ride/hihat/kick/snare/aux) — понятное наложение и per-lane
   mute/solo в редакторе.
5. **Явные crash-акценты как клипы** на нужных тактах вместо `crashType + everyN`.
6. **Velocity-envelope клетки** (опц. массив брейкпоинтов по тактам) сверх enum `dynamics`.
7. **Подключить `settingsComplexity`** к реальным настройкам или отказаться от фильтра по
   complexity в пользу явной расстановки.
8. **Упростить `MoleculeConditions`** — `barModulo`/`barRange` избыточны при явных спанах;
   оставить только доступность звуков (`requireRide` и т.п.).
9. **Playhead и loop-диапазон** в редакторе; клик по такту — превью с этого такта.
10. **Клип-пресеты / фразы** — сохранять типовые аранжировки для переиспользования.
11. **Двухуровневый zoom**: таймлайн клетки (такты) → таблица молекулы (доли/субдоли).

### UX Sketch (редактор клетки)

Грид: X = такты `1..length`, Y = лейны. Клип — горизонтальный блок на диапазоне тактов
(drag = сдвиг `startBar`, resize = `lengthBars`). Инспектор справа: **Velocity (общий)**,
dynamics, length/TS, «+ добавить клип». MVP редактора — числовые поля `startBar`/`lengthBars`
+ выбор лейна/mode; drag-and-drop отложить. Клик по клипу открывает `MoleculeTable`.
См. `output/files/cell-timeline-editor.excalidraw`.

### Migration Path

1. Добавить типы `DrumClip`/`ClipMode` и поля `arrangement?`, `velocity` в `DrumCell`.
2. Реализовать таймлайн-ветку в `assembleBar` + тесты; legacy-ветка нетронута.
3. Написать `convertLegacyCell()`: groove-пул → groove-клипы `[0,length]`; variation →
   клипы в позициях `k*variationEveryBars`; fill → клипы в fill-барах; crash → accent-клипы.
4. Прогнать конвертер по `DRUM_CELLS`, вручную довести «ручные» аранжировки (середина/
   вторая половина/брейки), A/B-сверить звук со старым путём.
5. Обновить `CellEditor` на таймлайн-редактор.
6. Депрекейтить legacy-поля и pool-ветку, когда все клетки на `arrangement`.

## Assumptions

- Все активные клетки — `4/4` (как сейчас); `5/4` учитывается через `timeSignature`, но не
  приоритет итерации.
- Клипы одного лейна не пересекаются по тактам (при пересечении — приоритет по порядку в
  массиве); enforcement — в редакторе.
- `barTicks = timeSignature[0] * 480`.
- Preview остаётся in-memory (без персистентности), как в текущем Конструкторе.
- Организмы сохраняются как уровень выбора клеток; таймлайн — внутри клетки.

## Risks

- High: рефактор `assembleBar` может дать регресс звука. Mitigation: двухрежимность +
  расширенные тесты идемпотентности + A/B-сверка legacy↔arrangement на всех клетках.
- Medium: вероятностные клипы ломают идемпотентность при неправильном сидировании.
  Mitigation: seed строго из `(organismSeed, barInOrganism, clipIndex)`, покрыть тестом.
- Medium: сложность таймлайн-редактора (drag/resize). Mitigation: MVP на числовых полях,
  drag-and-drop — отдельной итерацией.
- Medium: семантика `replace` (что именно глушить — лейн, весь грув, конкретные звуки).
  Mitigation: зафиксировать по лейнам; проверить на брейк-кейсах.
- Low: тайлинг `bars:2` на границе, если `length` не кратно 2. Mitigation: обрезать по
  спану/такту, тест на нечётной длине.
